# Autonomous AI Homelab Agent over SSH with Allowlist + Zoho Approval Gating: Feasibility and Implementation

## TL;DR

- **Yes, this is feasible today** by wrapping an existing agent rather than building from scratch. The strongest fit is **Claude Code’s `PreToolUse` hook** (which can `allow`/`deny`/`ask`/`defer` every Bash call and can call an HTTP endpoint) or, for fully open-source, **OpenCode’s `tool.execute.before` plugin hook** — both intercept shell tool calls deterministically before execution and cannot be bypassed by the model.
- **24/7 autonomous operation works via an external scheduler** (systemd timer or cron firing `claude -p`/`opencode run`/`openhands --headless`) — NOT the agents’ built-in schedulers, which are session-scoped and stop when the process exits. A small long-running “controller” daemon you write polls Netdata, spawns the agent per incident, and polls Zoho for approvals.
- **Authentication should live entirely outside the LLM context**: keep SSH keys, Netdata and Zoho tokens in the controller/daemon process (environment variables, a secrets manager via `apiKeyHelper`, or the agent’s secret registry that masks values), and have the agent connect through a pre-authenticated channel or a gating MCP/proxy server that holds the credentials — never pass secrets in prompts.

## Key Findings

### The architecture that works

The cleanest design separates three layers:

1. **A controller daemon (you write this, ~200 lines):** runs under systemd, polls Netdata for active alerts, and for each problem launches the agent in headless mode with the alert context. It also polls Zoho Desk for approval status and resumes/re-launches the agent when a command is approved.
1. **An off-the-shelf agent (Claude Code / OpenCode / OpenHands):** does the reasoning and proposes shell commands.
1. **A gating layer (hook or MCP/proxy server):** intercepts each shell tool call, checks it against the allowlist, executes it over SSH if allowed, and creates a Zoho Desk ticket + blocks if not.

All four agents you named can be wrapped; the differences are in *how* you intercept tool calls and *how* you handle the approval wait.

### 1. Claude Code — best-supported option

- **Hooks for tool calls: fully supported.** Claude Code has a `PreToolUse` hook that “runs after Claude creates tool parameters and before processing the tool call,” matches on tool name (`Bash`, `Edit`, `Write`, etc.), and can `allow`, `deny`, `ask`, or `defer` the call via a `hookSpecificOutput.permissionDecision` JSON field. The hook receives the full command on stdin as JSON (`tool_input.command`). Hooks “execute outside Claude’s control. The harness runs them, not the AI… Claude can’t skip hooks or work around them.” A `deny` works even under `--dangerously-skip-permissions`.
- **HTTP hooks (Feb 2026):** A `PreToolUse` hook can be type `http`, POSTing the tool-call JSON to your own server (e.g., `http://localhost:8080/hooks/pre-tool-use`) which returns the allow/deny decision. This is the natural home for allowlist + Zoho logic.
- **The `defer` decision is purpose-built for your approval flow:** for headless `claude -p` sessions, a `PreToolUse` hook can return `permissionDecision: "defer"`; the process exits with `stop_reason: "tool_deferred"` and the pending tool call preserved. Your controller surfaces it (creates the Zoho ticket), and when approved runs `claude -p --resume <session-id>`; the same hook fires again and returns `allow` with the answer in `updatedInput`. This is exactly the “create ticket → wait for human → execute” loop.
- **Headless/daemon mode: supported but not self-daemonizing.** `claude -p "<prompt>"` runs non-interactively, prints to stdout, exits. Combined with `--allowedTools`, `--output-format json`, `--max-turns`, and `--max-budget-usd` it is suitable for unattended runs. Its built-in `/schedule` and `CronCreate` tools are **session-scoped — they only fire while Claude Code is running and idle**, so they do NOT give you 24/7 unattended operation. For that, use a **systemd timer or cron** calling `claude -p` (Anthropic’s own docs and multiple guides show systemd `.service`+`.timer` units and cron entries as the recommended pattern).
- **Auth/secrets:** Claude Code reads `ANTHROPIC_API_KEY` from env, or supports an `apiKeyHelper` script that fetches keys at runtime from a secrets manager (1Password, Vault). Best practice is to keep secrets in environment variables so values “never appear in the chat” / context window, and to add deny rules like `Read(.env*)`, `Read(~/.ssh/**)`. Caveat: there are documented reports that Claude Code auto-loads `.env*` files and that `permissionDecision: "allow"` from a hook didn’t always suppress the native prompt in some recent versions — test your specific version.

### 2. OpenCode — best fully-open-source option

- **Allowlist is a first-class config feature.** OpenCode’s `permission` config gates the `bash` tool with **glob pattern matching**, e.g. `{"bash": {"*": "ask", "git *": "allow", "rm *": "deny"}}`, with last-matching-rule-wins semantics and per-agent overrides. This natively expresses your approved-commands allowlist.
- **Programmatic hook: `tool.execute.before`.** OpenCode’s plugin system (TypeScript/JS modules in `.opencode/plugin/`) exposes `tool.execute.before` — the documented equivalent of Claude’s `PreToolUse`. A plugin can inspect/mutate the bash command and **throw to block it**. This is where you’d run allowlist + Zoho ticket creation logic. (Note: blocking is via throwing; OpenCode does not currently have a built-in “pause and wait for external approval then resume the same call” primitive like Claude’s `defer`, so the wait must be handled by your controller re-invoking.)
- **Daemon/server mode: native.** `opencode serve` runs a **headless HTTP server** exposing an OpenAPI 3.1 spec; `opencode run "<task>"` is the one-shot headless CLI. The server can be protected with `OPENCODE_SERVER_PASSWORD` (HTTP basic auth). This is more daemon-friendly out of the box than Claude Code. Still session/process-oriented, so a systemd-managed controller is still the right approach for 24/7.
- **Auth:** API keys via env (`ANTHROPIC_API_KEY` etc.) or `/connect`; plugins can define custom auth providers (including OAuth with refresh). Multi-provider via models.

### 3. OpenHands (formerly OpenDevin) — strongest sandbox, weakest fit for “allowlist”

- **No native command allowlist; uses an LLM security analyzer + confirmation policy instead.** OpenHands assigns each action a risk level (LOW/MEDIUM/HIGH/UNKNOWN). The default `LLMSecurityAnalyzer` has the LLM self-annotate a `security_risk` field on each tool call (zero extra API calls). The `ConfirmRisky(threshold=HIGH)` policy auto-executes low-risk actions and pauses HIGH-risk ones in `WAITING_FOR_CONFIRMATION` until approved. You **can** write a `CustomSecurityAnalyzer` (subclass) that pattern-matches commands against your allowlist and returns HIGH for anything not approved — that’s the hook point. CLI flags: default asks per action; `--always-approve`/`--yolo`; `--llm-approve`. 
- **Caveat — known bugs:** there are open issues where the security analyzer “does not wait for user confirmation before… always runs the command,” the Invariant analyzer times out, and a CLI crash “LLM provided a security_risk but no security analyzer is configured.” Treat the gating as needing careful testing.
- **Daemon mode:** `openhands --headless -t "task"` (and `--json`), plus a Python SDK and `openhands serve` (Docker GUI server on :3000).  Headless mode “always approves” by default — must run in Docker with least-privilege mounts. 
- **Auth:** V1 SDK has a **Secret Registry** that stores secrets per-session, **late-bound (resolved only at exec time), masked in stdout/stderr as `<secret-hidden>`, encrypted at rest** — strong native support for keeping secrets out of the LLM context. The TerminalTool scans commands for secret keys and exports them as env vars.

### 4. Hermes (NousResearch) — model + its own agent framework

- **Hermes models** (Hermes 2 Pro / 3 / 4.3) are LLMs with a documented **function-calling standard** (`<tools>`, `<tool_call>`, `<tool_response>` tags).  They are not agents themselves, but vLLM/SGLang have built-in Hermes tool parsers,  so they slot into any framework (LangGraph, OpenAI Agents SDK, etc.).
- **`hermes-agent`** is NousResearch’s own open-source autonomous-agent framework wrapping these models. It is notably well-suited: it has **six terminal backends including SSH, Docker, Modal, Daytona**;  a **built-in cron scheduler with delivery to any platform** for “daily reports, nightly backups… running unattended”;  **MCP support**; and a security model where `execute_code` scripts run with **environment scrubbing (API keys/tokens/credentials stripped) and a tool whitelist**.  Notably, hermes-agent has an **open issue (#577) explicitly proposing to adopt OpenHands-style inline security-risk self-annotation**  — so a hard allowlist gate is not yet a built-in, you’d add it. Auth/config lives in `~/.hermes/config.yaml`; tools run in abstracted environments. 

### 5. General “secure shell agent” / gating patterns

- **LangGraph human-in-the-loop is the most mature wrap-anything pattern.** `HumanInTheLoopMiddleware` / the `interrupt()` primitive pauses the graph on a configured tool call, persists full state via a checkpointer (Postgres in prod), and resumes with `Command(resume=...)` after a human decides (approve/edit/reject/respond).  Because state is checkpointed, “interrupted threads… can be resumed many months later, on a different machine”  — ideal for waiting on a Zoho ticket. You’d drive a Hermes/other model through LangGraph and put the allowlist+Zoho logic in the interrupt handler.
- **OpenAI Agents SDK** also has HITL: `needs_approval`/`require_approval` on shell, MCP and function tools, pausing on `RunResult.interruptions` and resuming after `state.approve()/reject()`. 
- **Gateway products:** there is emerging prior art for a “control plane / zero-trust gateway for AI agents.” The clearest example is **Peta.io** (Dunia Labs Inc., founded mid-2025), branded “1Password for AI Agents,” which “combines an MCP gateway with an agent-facing vault and policy engine”  that “stores all external API credentials… server-side in an encrypted vault and issues short-lived tokens to agents instead of ever exposing raw API keys.” Related named MCP gateways include **TrueFoundry** and **MintMCP** (the latter described as “the industry’s first SOC 2 Type II certified MCP platform”). Useful as design references — you don’t need one to build this.

### 6. MCP server as the gating layer — viable and clean

A custom MCP server **can** be the gate. Approaches:

- **MCP Elicitation** — “newly introduced” in the official **MCP specification 2025-06-18** — “provides a standardized way for servers to request additional information from users through the client during interactions.”  It uses the `elicitation/create` JSON-RPC method, validates input via JSON Schema, and the user’s response returns one of three actions: **`accept`, `reject`, or `cancel`** — the standardized HITL primitive underlying the approval gating documented by Cloudflare Agents, Agno, and the OpenAI Agents SDK (`require_approval: "always"`).  **Important constraint:** the spec states “Servers MUST NOT use elicitation to request sensitive information,” so use elicitation for the approve/deny *decision*, not for passing credentials.
- **Practical pattern for your use case:** expose a single `run_shell_command` MCP tool from your own server. Inside it: check the allowlist; if approved, SSH to the homelab box (credentials held by the MCP server process, never the LLM) and run it; if not approved, create a Zoho ticket and either block with an explanatory message or (with a durable workflow like Temporal/Cloudflare Workflows `waitForApproval()`) hold until approved. This keeps both the allowlist AND all secrets entirely server-side, and works with any MCP-capable agent (Claude Code, OpenHands, OpenCode, hermes-agent all support MCP). The note from practitioners: don’t hardcode the “dangerous/allowed” list inside the server — externalize it to a policy file ops can edit. 

### 7. Netdata Cloud API — yes, pollable for active alerts

- **Agent (local) endpoint, confirmed:** `GET http://NODE:19999/api/v1/alarms?all` “will return the alerts currently in WARNING or CRITICAL state.”  Incremental history via `/api/v1/alarm_log?after=<UNIQUEID>`. Local API is unauthenticated from localhost by default.
- **Cloud REST API:** Base `https://app.netdata.cloud/api/v2/…` and `/api/v3/…`, authenticated with `Authorization: Bearer <token>`;  tokens are generated in Netdata Cloud **User Settings → API Tokens**. Per Netdata’s REST API docs: “Choose `scope:all` to grant the token access to all Cloud API endpoints… The token generates and displays once for security reasons. Save it securely — you’ll need to generate a new token if you lose it.”  (Note: the docs explicitly state non-expiry for *TV Mode* tokens — “TV Mode tokens do not expire automatically”  — so verify the never-expire behavior for standard API tokens rather than assuming it.) Documented working calls include `GET /api/v2/spaces`,  `/api/v2/nodes`, and POST `/api/v3/spaces/{SPACE_ID}/rooms/{ROOM_ID}/data`.
- **Active-alerts endpoint:** the Agent/Parent OpenAPI spec documents **`GET /api/v3/alerts`** (“current status of all alerts across all nodes” — i.e. currently raised) and a separate **`GET /api/v3/alert_transitions`** (history/log); there is also a deprecated **`GET /api/v2/alerts`** (“returns all running alerts, per node”). The Cloud equivalent mirrors the data-endpoint pattern: **`GET https://app.netdata.cloud/api/v3/spaces/{SPACE_ID}/rooms/{ROOM_ID}/alerts`**. **Caveat:** Netdata’s own docs state the Cloud API is “a work in progress”  and “not exposing the stable API,” and they do not print this exact Cloud alerts path verbatim — confirm it in the logged-in interactive Cloud API explorer before relying on it. For history use `/alert_transitions`; for current state use `/alerts`. Netdata also offers an MCP server (`https://app.netdata.cloud/api/v1/mcp`, `scope:mcp` token) that gives an AI “real-time visibility into active and raised alerts” — a turnkey alternative to manual polling.

### 8. Zoho Desk API — yes, supports create + status polling

- **OAuth 2.0** (client ID/secret from Zoho Developer Console/API Console → access + refresh tokens). Per Zoho’s official OAuth token-limits docs: “At a time, a maximum of **20 active refresh tokens** can be stored by a client per user,” and “a maximum of **10 active access tokens** can be stored by a client per refresh token” — so a long-running daemon must reuse/refresh tokens rather than minting new ones each cycle. Calls pass `Authorization: Zoho-oauthtoken <token>`  and an `orgId` header. Base: `https://desk.zoho.com/api/v1/…`.
- **Create ticket:** `POST /api/v1/tickets` (supports `customFields`). **Get/poll a ticket:** `GET /api/v1/tickets/{id}` (single record = 1 API credit).  **Update status:** `PATCH`/update on the ticket, or use Blueprint **transitions** (`/api/v1/tickets/{ticketId}/transitions/{transitionId}/perform`). **List:** `GET /api/v1/tickets`. **Webhooks** can push ticket-update events instead of polling.
- **Approval modeling:** simplest is a **custom field** (e.g. `cf_approval = approved/denied`) the human sets, which your controller polls via `GET /api/v1/tickets/{id}` and reads from `customFields`. Zoho Desk also has a native Approvals module, but a custom field or ticket status is the lowest-friction approach.

## Details: recommended concrete implementation

**Pick Claude Code if** you can use a commercial model and want the least custom code — its `PreToolUse` HTTP hook + `defer`/`--resume` gives you the approval-wait loop almost for free. **Pick OpenCode if** you require fully open-source/self-hosted — `permission` allowlist + `tool.execute.before` plugin + `opencode serve`. **Pick hermes-agent if** you want an open-source agent with native SSH backends and a built-in cron scheduler and are willing to add the allowlist gate. **Use LangGraph** if you want maximum control over the pause/resume-on-approval state machine around an open model.

**Reference flow (Claude Code variant):**

1. **systemd timer** (e.g. `OnCalendar=*:0/5`) fires a controller script every 5 min (use a timer/daemon, not Claude’s session-scoped `/schedule`).
1. Controller calls Netdata Cloud `GET /api/v3/spaces/{space}/rooms/{room}/alerts` (Bearer token in controller env) — or the local `GET /api/v1/alarms?all`. If WARNING/CRITICAL alerts exist, it builds a prompt with the alert context.
1. Controller launches `claude -p "<alert context + task>" --allowedTools "Bash" --output-format json --max-turns N`, with a `PreToolUse` Bash hook configured as an **HTTP hook** pointing at the local gating service.
1. **Gating service** receives each proposed command. If it matches the allowlist → returns `permissionDecision: allow`; the command is executed **over SSH using a key the gating service/controller holds** (not in the prompt). If not on the allowlist → it creates a Zoho Desk ticket (`POST /api/v1/tickets` with the command in a custom field) and returns `permissionDecision: deny` (or `defer` for headless resume).
1. Controller **polls Zoho** `GET /api/v1/tickets/{id}` for the approval custom field. On approval, it re-launches `claude -p --resume <session-id>`; the hook now returns `allow` and the command runs. Optionally add the now-approved command to a persistent allowlist.

**Keeping auth out of the LLM:** SSH private key, Netdata token, and Zoho OAuth tokens all live in the **controller/gating-service process environment** (or a secrets manager fetched via `apiKeyHelper`). The agent never sees them: either (a) the gating service executes the SSH itself, or (b) the agent SSHes via an already-loaded `ssh-agent`/control socket with no key material in context. OpenHands’ Secret Registry and Hermes’ env-scrubbing achieve the same masking if you use those agents.

## Recommendations

1. **Prototype with Claude Code first** (fastest path to the approval loop): implement the `PreToolUse` HTTP hook + allowlist + Zoho ticket creation, validate the `defer`/`--resume` cycle end-to-end on a non-critical alert. **Threshold to switch to OpenCode:** if licensing/cost or a hard “fully open-source/local-model” requirement applies.
1. **Run the controller as a systemd service + timer**, not via the agent’s internal scheduler. Cap every agent run with `--max-turns` and a budget/cost ceiling, and log all proposed and executed commands.
1. **Put the SSH execution and all credentials in the gating layer (HTTP hook service or a custom MCP server)**, never in prompts. Strongly prefer the **MCP-server-as-gate** design if you want to swap agents later, since all four support MCP. Remember MCP elicitation is for the approve/deny decision only — never route secrets through it (the spec forbids it).
1. **Model approvals as a Zoho custom field or Blueprint transition**, and prefer **webhooks over polling** once the prototype works to reduce API-credit usage and latency. Reuse one refresh token (you’re capped at 20 per user) rather than re-authenticating each cycle.
1. **Harden against known caveats:** pin agent versions and test that hook `deny`/`allow` actually fire on your version (Claude Code had a regression where `allow` didn’t suppress the prompt; OpenHands’ security analyzer has had wait/crash bugs); add `Read(.env*)`/`Read(~/.ssh/**)` deny rules; run the homelab target with a least-privilege SSH user.
1. **Verify the exact Netdata Cloud alerts path** in the interactive Cloud API explorer (it’s officially “work in progress”); fall back to the confirmed local Agent endpoint `GET /api/v1/alarms?all` if needed.

## Caveats

- **Session-scoped vs. true daemon:** None of these agents are themselves long-lived 24/7 daemons in the way the use case implies; the “always-on” property comes from your systemd-managed controller. Claude Code’s `/schedule`/`CronCreate` and Hermes’ built-in cron are the closest native schedulers, but Claude’s only fires while a session is open.
- **Allowlist enforcement is only as strong as the gate.** A model can chain commands (`a && b`), write a script then execute it, or use a different tool to bypass a narrow matcher — a documented Claude Code issue showed `Edit` blocks bypassed via the `Bash` tool. Enforce at the SSH-execution chokepoint (parse/validate the actual command string server-side), not just via prompt or a single tool matcher, and prefer denylist+allowlist defense-in-depth plus a least-privilege remote user.
- **The Cloud Netdata alerts endpoint is inferred, not officially documented verbatim**; the API is explicitly pre-stable. The standard-API-token non-expiry claim is also unverified (docs only confirm non-expiry for TV Mode tokens).
- **OpenHands gating has open reliability bugs**; Hermes lacks a built-in hard allowlist (only a proposal). Both need extra work versus Claude Code/OpenCode.
- **Cost/runaway risk:** an autonomous loop that re-launches per alert can rack up token spend or thrash; always set `--max-turns`, budget caps, and alert-deduplication (Netdata Cloud already deduplicates) so one flapping alert doesn’t spawn endless agent runs.
